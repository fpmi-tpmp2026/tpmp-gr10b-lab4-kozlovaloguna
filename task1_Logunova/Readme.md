## Главная страница 

Данный проект посвящён созданию базы данных «Абитуриент» и выполнению операций SQL (CRUD, фильтрация, экспорт), а также моделированию системы с помощью диаграмм UML.

### Задание 1 (постановка задачи)

**Цель:** Разработать структуру базы данных «Абитуриент», содержащую следующие поля:
- `id` (уникальный идентификатор)
- Фамилия, имя, отчество
- Пол, национальность
- Дата рождения
- Домашний адрес (почтовый индекс, страна, область, район, город, улица, дом, квартира)
- Оценки по ЦТ
- Проходной балл

**Требуемые операции SQL:**
1. Создание таблицы (`CREATE`)
2. Вставка данных (`INSERT`)
3. Выборка с сортировкой по `id`, по имени, вывод последних 5 строк (`SELECT ... ORDER BY ... LIMIT`)
4. Фильтрация по `id=5` (`WHERE`)
5. Поиск по маске: фамилия начинается на определённую букву (`LIKE`)
6. Переименование таблицы (`ALTER TABLE ... RENAME`)
7. Обновление данных (`UPDATE`)
8. Удаление строк по `id` и по фамилии (`DELETE`)
9. Экспорт базы в файлы `.sql` и `.csv`
10. Удаление таблицы (`DROP TABLE`)

**Дополнительные запросы (Упражнение 3.2):**
- Вывести названия и описания всех категорий расходов.
- Вывести названия магазинов, даты покупок и суммы покупок, отсортированные по убыванию суммы.

## 2. Диаграммы UML 

Ниже представлено описание каждой диаграммы, построенной для данной системы.

### 2.1. Диаграмма вариантов использования (Use Case Diagram)

**Что делает:**  
Показывает, какие функции система предоставляет разным типам пользователей (акторам). Выделяет границы системы и взаимодействие внешних сущностей с ней.

**В данной работе диаграмма отражает:**
- **Акторы:** Оператор ПК (вносит, редактирует, удаляет данные), Абитуриент (только просматривает информацию и расходы), Система (автоматический экспорт).
- **Основные варианты использования:**
  - Добавление / редактирование / удаление абитуриента.
  - Просмотр списка абитуриентов.
  - Фильтрация по проходному баллу.
  - Поиск по фамилии (с использованием `LIKE`).
  - Экспорт данных в `.sql` и `.csv`.
  - Просмотр категорий расходов и покупок с сортировкой.

**Текстовый сценарий (для варианта «Добавить абитуриента»):**
1. Оператор открывает форму добавления.
2. Система отображает пустые поля.
3. Оператор вводит данные (ФИО, адрес, оценки).
4. Система проверяет корректность форматов.
5. При успехе система выполняет `INSERT` и выводит сообщение об успехе.
6. При ошибке — запрашивает повторный ввод.


### 2.2. Диаграммы деятельности (Activity Diagram)

**Что делают:**  
Описывают алгоритм выполнения операций шаг за шагом. Показывают ветвления (условия), циклы и параллельные потоки.

**В данной работе построены две диаграммы:**

#### Диаграмма 1: «Добавление абитуриента»
- **Что показывает:** Последовательность действий от момента выбора пункта меню до сохранения записи.
- **Логика:** Оператор вводит данные → система валидирует → если ошибка → повтор ввода → если всё верно → `INSERT` → сообщение об успехе.

#### Диаграмма 2: «Экспорт данных»
- **Что показывает:** Процесс выгрузки базы во внешние форматы.
- **Логика:** Оператор выбирает формат (SQL или CSV) → система формирует запрос `SELECT *` → преобразует результат в нужный формат → сохраняет файл на диск → уведомляет об успехе.


### 2.3. Диаграмма классов (Class Diagram)

**Что делает:**  
Показывает статическую структуру данных: сущности (классы), их атрибуты, методы и связи между ними.

**В данной работе диаграмма включает:**

1. **Класс `Applicant` (Абитуриент)**
   - **Атрибуты:** `id`, фамилия, имя, отчество, пол, национальность, дата рождения, адрес (разбитый на поля: индекс, страна, область, район, город, улица, дом, квартира), оценки ЦТ, проходной балл.
   - **Методы (операции):** `create()` (вставка), `update()` (обновление), `delete()` (удаление), `selectAll()` (вывод всех), `selectById()` (поиск по ID), `selectLike()` (поиск по маске).

2. **Класс `Spending` (Покупка)**
   - **Атрибуты:** `id`, название магазина, дата покупки, сумма.
   - **Метод:** `selectOrderByAmountDesc()` — выборка с сортировкой по убыванию суммы.

3. **Класс `ExpenseCategory` (Категория расходов)**
   - **Атрибуты:** `id`, название категории, описание.
   - **Метод:** `selectAll()` — получение всех категорий.

**Связи между классами:**
- Один абитуриент может совершить много покупок (связь 1 : *).
- Одна покупка относится к одной категории расходов (связь * : 1).


### 2.4. Диаграммы последовательности (Sequence Diagram)

**Что делают:**  
Показывают обмен сообщениями между объектами во времени. Уточняют, как именно происходит взаимодействие актора, приложения и базы данных.

**В данной работе построены две диаграммы:**

#### Диаграмма 1: «Вставка нового абитуриента»
- **Участники:** Оператор, Приложение, База данных SQLite.
- **Ход взаимодействия:**
  1. Оператор отправляет заполненную форму приложению.
  2. Приложение валидирует данные.
  3. Приложение отправляет SQL-запрос `INSERT` в базу данных.
  4. База данных сохраняет запись и возвращает `id` новой записи.
  5. Приложение выводит оператору сообщение об успехе.

### 2.5. Диаграмма компонентов (Component Diagram)

**Что делает:**  
Показывает физическую структуру системы: какие программные компоненты существуют, как они распределены и как взаимодействуют друг с другом.

**В данной работе диаграмма включает три слоя:**

1. **Клиентская часть (Client Layer)**
   - Компоненты: консоль SQLite (для ручного ввода команд), скрипты экспорта (например, на Python или Bash).

2. **Слой бизнес-логики (Business Logic Layer)**
   - Компоненты: CRUD-менеджер (обрабатывает запросы на создание, чтение, обновление, удаление), валидатор данных (проверяет корректность перед записью), генератор отчётов (формирует CSV и SQL-дампы).

3. **Слой хранения данных (Data Layer)**
   - Компоненты: файл базы данных SQLite (`applicants.db`), папка с экспортированными файлами (`backup.sql`, `data.csv`).

**Связи:**
- Клиент обращается к бизнес-логике.
- Бизнес-логика читает/пишет данные в слой хранения.
- Генератор отчётов экспортирует данные из слоя хранения во внешние файлы.
